# Requirements: Insert Events Feature

## Intent Analysis

**Goal**: Allow users to manually insert conversation events into a specific AgentCore Memory session via a user-friendly UI, using a structured form with role dropdowns and message inputs.

**Type**: New user-facing feature — frontend UI + frontend API client (backend endpoint already exists)

**Scope**: Frontend only (backend `POST /memories/{memory_id}/events/bulk` is complete and production-ready)

---

## Functional Requirements

### FR-1: Entry Point
- The feature MUST be accessible via a **modal or dialog** (not a separate page)
- The modal MUST be reachable from the **Memory Detail view** (where sessions are listed for a memory)
- The trigger is a clearly labeled button (e.g., "Insert Events")

### FR-2: Actor ID Input
- The modal MUST include a **text input field** for `actor_id`
- The field MUST be required — submission is blocked without it
- No pre-filling — the user types it manually

### FR-3: Session ID Input
- The modal MUST include a **text input field** for `session_id`
- The field is **optional** — if left blank, a new session is auto-generated by the backend
- Placeholder text should communicate this behavior (e.g., "Leave blank to auto-generate")

### FR-4: Event Builder — Structured Form
- The modal MUST allow the user to compose **one or more events**
- Each **event** contains one or more **messages**
- Each **message** has:
  - A **role dropdown** with options: `USER`, `ASSISTANT`, `SYSTEM`, `TOOL`
  - A **text input** (or textarea) for the message content
- Users MUST be able to:
  - Add a new message to the current event
  - Remove a message from an event
  - Add a new event (new turn in conversation)
  - Remove an event
- Default state: 1 event with 2 messages (USER + ASSISTANT) as a starting template

### FR-5: Submission
- A **Submit / Insert** button sends the data to `POST /memories/{memory_id}/events/bulk`
- On success:
  - The modal closes
  - The session list refreshes to show the newly created/updated session
  - A success notification is displayed
- On error:
  - The modal stays open
  - An inline error message is displayed

### FR-5b: Loading State During Submission
- While the request is in-flight, the **Submit button** MUST show a loading/spinner state and be disabled
- The **Cancel button and close (X)** MUST also be disabled during submission to prevent accidental dismissal
- All form fields MUST be disabled during submission to prevent edits mid-flight
- No full-page spinner — loading state is scoped to the modal controls only

### FR-6: Cancel / Close
- A **Cancel** button and/or close (X) action closes the modal without submitting
- No confirmation required for cancel

## Non-Functional Requirements

### NFR-1: UI Consistency
- The modal MUST use **Cloudscape Design System** components — consistent with the rest of the app
- Specific components to use:

| Component | Package | Purpose |
|---|---|---|
| `Modal` | `@cloudscape-design/components/modal` | Main dialog container |
| `Form` | `@cloudscape-design/components/form` | Form wrapper with submit action |
| `FormField` | `@cloudscape-design/components/form-field` | Label + validation error for each field |
| `Input` | `@cloudscape-design/components/input` | `actor_id` and `session_id` text fields |
| `Textarea` | `@cloudscape-design/components/textarea` | Message content (multi-line) |
| `Select` | `@cloudscape-design/components/select` | Role dropdown (`USER`, `ASSISTANT`, `SYSTEM`, `TOOL`) |
| `Button` | `@cloudscape-design/components/button` | Submit (with `loading` prop), Cancel, Add/Remove message, Add/Remove event |
| `SpaceBetween` | `@cloudscape-design/components/space-between` | Vertical/horizontal spacing throughout |
| `Box` | `@cloudscape-design/components/box` | Layout primitives |
| `Container` | `@cloudscape-design/components/container` | Card-like wrapper per event block |
| `Header` | `@cloudscape-design/components/header` | Section title per event (e.g. "Event 1") |
| `Alert` | `@cloudscape-design/components/alert` | Inline error message on submission failure |
| `Flashbar` | `@cloudscape-design/components/flashbar` | Success toast after insertion (consistent with app patterns) |

### NFR-2: Usability
- The form MUST be intuitive without documentation
- Role dropdown MUST show human-readable labels (not just raw strings)
- Adding/removing messages and events should feel natural (inline add/remove buttons)

### NFR-3: Validation
- Client-side validation before submission:
  - `actor_id` is required
  - At least 1 event with at least 1 message required
  - Each message must have non-empty content
- Validation errors displayed inline

### NFR-4: Frontend API Client
- A `bulkSeedEvents()` function MUST be added to `frontend/src/lib/api/events.ts`
- TypeScript types for the request and response MUST be defined in `frontend/src/types/index.ts`

---

## Out of Scope

- Backend changes (endpoint already complete)
- Raw text/paste parsing (user types structured data directly)
- Editing existing events (insert-only)
- Timestamp control by the user (auto-handled by backend)
- Batch import from files

---

## Acceptance Criteria

| # | Criteria |
|---|---|
| AC-1 | "Insert Events" button appears on the Memory Detail page |
| AC-2 | Modal opens with actor_id (required), session_id (optional) fields |
| AC-3 | User can add/remove messages within an event using role dropdown + content input |
| AC-4 | User can add/remove multiple events |
| AC-5 | Submitting calls `POST /memories/{memory_id}/events/bulk` with correct payload |
| AC-6 | On success: modal closes, session list refreshes, success toast shown |
| AC-7 | On error: modal stays open with error message |
| AC-7b | While submitting: Submit button shows spinner + is disabled; Cancel/close and all fields are also disabled |
| AC-8 | All UI uses Cloudscape components |
| AC-9 | `bulkSeedEvents()` is typed and exported from the API client module |
